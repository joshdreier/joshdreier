Developers Guide to Web Content Accessibility Guidelines
(WCAG) 2.0 Level A/AA

Page titles(2.4.2)

  • All pages must have a <title></title>.
  • The title must describe topic or purpose of the webpage.
  • Multiple webpages on the site should not have the same title when they have different content or purpose. When a page loads, a visually impaired screenreader user first hears the page title. If they have already encountered that title, they may assume that they have already seen this page, and navigate away from it.
  • It is best practice to front load the most important and page specific part of the title – that is, start with most specific description the content or purpose of the current webpage, followed by a site section name and finally the site name. That way the user doesn't need to hear text that is repeated as the start of several web pages, and immediately knows that they have reached the page they expect.
  • Usually the main <h1> content heading on a page matches or reflects the <title> text.
  • Avoid using pipes in title (|), as screen reader will read out "vertical line."

Semantic markup(4.1.1)

  • All markup elements should have complete start and end tags
  • Elements should be nested according to their specifications
  • Elements should not contain duplicate attributes, and any IDs should be unique.
  • Maximize compatibility with current and future user agents, including assistive technologies.

Headings and page structure (1.3.1, 2.4.1, 2.4.6, 4.1.2)

  • There must be a mechanism for users to bypass repeated blocks of content on a webpage – usually a mechanism to bypass the topmost <header> region to reach the <main> content is sufficient.
  • There should be only one <h1> heading on a page, heading up the main content.
  • Page regions should be marked up into semantic regions.
  • The heading structure of a web page must be hierarchical. In practice this means looking at different regions on a webpage, and creating heading structures within each one.The main content region is headed up by a descriptive <h1> heading, then main content sections and sub sections marked up using <h2>s, <h3>s, etc. Ideally the <h1> should be the first content in the region, but a small amount of content before it, such as a breadcrumb trail is permissible.
  • Multipart headings (3.1.2) (items with seemingly 2 different headings) should not be structured into 2 separate headings. Example of correct way: <h3><span class="top">Latest News</span>Stock Market Up</h3>
  • Paragraphs in the content must be marked up semantically using <p> elements.

Content order (1.3.1, 1.3.2)

Content order is important. For instance, in the HTML markup, headings should precede their associated content, even if CSS is used to reorder or arrange content visually on the webpage. Ask yourself: if you were to switch off CSS, would the main content form a logically structured document? If not, then the order of the content in the HTML mark-up must be rearranged to make it logical.

Lists (1.3.1, 1.3.2)

Modules with items that present themselves in a horizontal or vertical array should be marked up as a semantic unordered list (<ul>). If there is a certain order to the list, such as breadcrumbs, an ordered list (<ol>)can be used.

Visible focus outlines, keyboard tabindex (1.4.3, 2.1.1, 2.1.2, 2.4.3, 2.4.7, 3.2.1)

When tabbing through the site, all items that are focusable (<a>, <button>, <input>, <select>, <textarea>, <iframe>, etc.) are part of the tab index, and need to have a bold indicator outline meeting the 3:1 color contrast ratio requirement.

  • Use "tabindex: '-1'" to remove focusability from elements
  • Use "tabindex: '0'" to add focusability to elements
  • A CSS mixin for the outline is useful to apply options to certain elements that need outlines with different colors or positioning. box-shadow can be used as the outline for more control.
  • Mouse users do not need to see the focus outline. JS can be used to listen for mouse vs. keyboard clicks, and only show outline for keyboarders.
  • Confirm keyboard focus is never locked or trapped at one particular page element. The user can navigate to and from all navigable page elements using only a keyboard.

Hiding content off-screen (2.4.6, 2.4.7)

Occasionally, it is useful if content were present in HTML mark-up, but hidden from view. This is so visually impaired screenreader users have access to information that might not be visually apparent otherwise, but the content does not visually clutter the webpage. Use a global CSS class or mixin to hide the content offscreen.

Focusable items, such as <button>, that are hidden off-screen, should become visible on :focus. Here is example CSS, that hides a button offscreen unless it is focused:

button {
    &:not(:focus) {
        @include visually-hidden;
    }
}

Hiding content from screen readers and assistive technologies (2.1.1)

  • The best way to hide content from assistive technologies is to use the CSS property, "display: none;"

  • If "display: none;" cannot be used for some reason, the next best option is to use "aria-hidden='true'" to hide elements from assistive technologies. This can be useful for carousel items that are offscreen, for instance.
  • If "aria-hidden='true'" is used instead of "display: none," make sure that all focusable elements inside the container have been removed from the tabindex. This can be done using by setting "tabindex='-1'"

Button controls for same page interaction/navigation (2.1.1, 2.4.3)

Ensure that all focusable controls have the correct and expected roles. In general,

  • Controls which are links to other webpages must have a link role. They should be marked up using the native hyperlink element (<a>). If this is not possible, then they should be given an ARIA role=”link” attribute.
  • Controls which have a same page action or submit a form, must be marked up as a <button>. Generally, controls that have a same page action, should have a button role. This is best achieved using the native <button> element. If this is not possible, then they should be given an ARIA role=”button” attribute. This includes:
    • Accordion controls
    • Display navigation controls (such as hamburger menu controls)
    • Open and close modal controls
    • Video controls (play, pause, stop, etc)
    • Next and previous buttons for image viewers.
    • Carousel controls, including next and previous buttons, and individual slide detector buttons
    • Slide show controls
    • Main menu hamburger button

Exceptions

For menu links, in a navigation menu, which when activated display a dropdown menu or other navigation panel. These technically should have button roles, but are so well known, that hyperlinks will do if the hyperlinks have aria-expanded attributes which are managed.

If a hyperlink or button is marked up using a generic element, such as a div or a span, then these,

  • Must be given an appropriate ARIA role
  • Must be added to the tab order using tabindex=”0”, so as to be focusable for keyboard-only users
  • Must have, in addition to click events, keyboard events defined, so users can activate them using the enter key, and/or the space bar key.

"Learn more" or "Read More" links (2.4.4)

Links such as “Read more” or “Learn more” on their own don’t provide a complete description of their destination. The text preceding them provides a content which makes their destination clear, and this context must be programmatically associated with the link.

  • The destination of the link must be clearly and unambiguously described, and must be programmatically determinable
  • As they are links to new pages, they must have a link role (eg, using an <a> element); they must not have button roles, even if they visually appear button-like.

From the visual context, it is apparent what the destination of a “More” type of link is. However, this visual context, which completes an accurate and complete description of the destination of the link, must also be programmatically determinable. Without this, for visually impaired screen readers the destination of the link would not be clear from the text alone. This will be exasperated when there are multiple “more” type links on a page, as the links would have identical label text, but different destinations. The most reliable solution is to design the component so that the visible descriptive text is in the same structural element as the link, so can be accessed by a screenreader without the user having to move away from the link itself. This is what is meant by programmatically determinable.

The structural containing elements suitable to wrap around the link AND descriptive content to make the purpose of the link clear include:

  • <p>, where the link is part of the paragraph that contains the description
  • <li>, where the link is part of a list item that provides the context

The above solution is considered most reliable because it doesn't depend on content editors having to change text that they cannot see other than in code view.

There are many other possible solutions; one of the first two solutions (title attribute, or hidden text) would probably be easiest

  • Using a title attribute
  • Using hidden text within the <a>
  • Using an aria-describedby attribute
  • Using an aria-label attribute

Images (1.1.1, 1.3.1)

Text alternatives for images

Informational images: Images that convey information must have accessible text alternatives,

  • If the image does not need a visible caption, then it can be served in an <img> element, with an descriptive alt attribute.

  • If the image has a visible caption, it can be served up by a <figure>/<figcaption> element set. The <img> should be given an empty alt attribute, and the caption marked up as a <figcaption>

    <figure>
    <img alt=”” . . . />
    <figcaption>Caption text</figcaption>
</figure>

    A less desirable option would be to alert the screenreader to the presence of an image, and from that they can infer the text immediately following it is its caption. Ideally it should be grouped within an <aside> element, which would alert the user that this is supplementary information, and allow them to move to the end of the <aside>, and on with the article text.

    <aside>
    <img alt=”Image:” . . . />
    <span>Caption text</span>
</aside>

Decorative images

A decorative image must be hidden from assistive technologies. This can be done in the following ways,

  • Preferably, serving up the image using a CSS background-image; which will be automatically hidden from screen readers.
  • If not, then serving up the image using a CSS pseudo element. But as this may be voiced by a screenreader, the element must be hidden from screen readers using the aria-hidden attribute,
    <span class=”icon” aria-hidden=”true”></span>
  • Or serving up the image in an <img> element, with an empty alt attribute

Graphic symbols (eg, greater than symbol) (1.1.1)

The arrow graphic should be rendered in such a way to be hidden from screen readers and other assistive technologies. (If it is a character in the HTML mark-up, screen readers will announce the link as “Learn more greater than”). Hiding the symbol can be achieved by one of the following methods,

  • Serve up the symbol using a CSS background-image rule.
  • Serve up the symbol using a CSS content rule, but hidden from assistive technologies using the aria-hidden attribute.
  • Using a greater than symbol in the HTML markup, but hiding it with an aria-hidden attribute
    <span aria-hidden=”true”>&gt;</span>

Asides with quoted text (1.3.1, 1.3.2)

Similar to images in the article text, sometimes a quote box is pulled out of the article and inserted in the flow of the article’s content. If this text were undifferentiated from the rest of the content, screenreader users might be confused: text would be for some reason read out twice, and where it occurs in the flow of text may be illogical.

A solution is, again, to use an <aside> element. Screenreader users encountering it will be alerted that this content is outside the flow of the article text, and choose whether to read it, or jump over it and continue with the article text.

<aside>
    <blockquote>
        text text
    </blockquote>
</aside>

Tabbed content (2.1.1)

  • Roles and states must be defined for the tab list, individual tabs, and tab panels. States must be managed
  • Keyboard-only users must be able to interact with tabs in the expected way.

Badly marked up tabs and tab panels can cause havoc for assistive technology users. So if it can't be done properly, then find an alternative way to display this type of content.

To get it right, all states, roles and associations must be defined using ARIA attributes including,

  • the role attribute: tabs should have role="tab", and the tab panels themselves role="tabpanel".
  • The relationships between tabs and tab panels can be defined using aria-controls (on tabs) and aria-labelledby (on tab panels) attributes.
  • The state of each panel (displayed or hidden) is defined by an aria-selected attribute on the tab, which must be managed (true or false).

All keyboard-only users should be able to tab to the currently selected tab, and then navigate through them using the left and right arrow keys.

A good fully working example of an accessible tab panel system has been developed by Nicolas Hoffmann: http://a11y.nicolas-hoffmann.net/tabs/#robust_accessible

Modal windows (2.1.1, 2.4.7, 3.2.1)

There are many ways to technically implement modal dialogs, but in general the required mark-up and behavior to achieve an accessible implementation is as follows,

  • The open modal control must have a button role
  • When the modal is opened, focus must be managed (moved to modal)
  • The modal must be able to be closed, and when it is, focus must be managed (moved back to triggering element)
  • When open, users interaction must be confined to the contents of the modal. This includes screenreader users.

An example of an accessible implementation of a modal is Greg's Incredible Accessible Modal Window.

The open modal control role

The control has a same page action, and must have a button role. This is best achieved by using the native <button> element.

Managing focus when opening:

When the modal is opened, focus should be moved to either the modal’s

  • Close button
  • The first heading in the modal
  • the modal container itself
  • The first or only form field in the modal

Closing modal dialog:

The modal dialog must be able to be closed. While it is good practice for the escape key to close the modal, it is not sufficient on its own. A close <button> must be present.The position of the keyboard focus before the modal popup was opened should be saved, so when the modal is closed, focus can be returned to that location.

Constraining screen readers to content of dialog

The modal dialog must constrain the screenreader user within the modal dialog content so that the user can’t accidentally interact with the page behind the dialog while it is open. This can be achieved by wrapping the entire page’s contents in a container with an aria-hidden attribute. Eg, <div aria-hidden=”true”>.

Informing screenreaders that focus has moved

When a modal is opened, and focus has been moved, screenreader users should be informed that focus has been moved and they have reached the content. A convenient way of achieving this is to make sure the modal has a main <h1> heading, and that this is marked up as an ARIA live region. When live regions are revealed in the DOM, or their contents changed, those contents are announced by screenreaders. So for instance,

<h1 aria-live="assertive">
    Modal Media Gallery
</h1>

If the heading does not need to be visually displayed, it can be hidden off-screen using CSS.

Carousels (2.1.1, 2.2.2)

Digital Pulp recommends using the "Slick" Carousel plugin library, as it provides the most comprehensive accessibility features out of the box: http://kenwheeler.github.io/slick/

For sighted keyboard-only users:

  • All carousel controls must be in the tab order, and be reachable and operable using only the keyboard.
  • All the controls must have visible focus indicators, so users can see which controls have received focus. These must be accessible - see “Visible focus outlines" section above.
  • The controls and slides must have a logical sequence. For instance: Previous, Next, followed by individual slide pickers (these may be dots, or thumbnails).
  • Users must not be able to tab to any focusable controls in slides which are hidden from view. Ideally hidden slides should be removed from the DOM using display:none. If "display: none;" cannot be used for some reason, the next best option is to use "aria-hidden='true'" and tabindex="-1" within the slides to hide elements from assistive technologies.

The above requirements can be easily tested by unplugging the mouse and attempting to operate the carousel using only the keyboard. Meeting these requirements will probably mean the carousel is accessible for sighted keyboard-only users. However, they also need to be made accessible for users relying on access technologies such as screen readers, which visually impaired users may be using.

  • All controls must be marked up semantically with the correct button roles (next, previous and individual slide selectors).
  • All next and previous controls, and the individual slide selectors must have accessible text labels (that is, text alternatives for symbols) that describe their purpose. Without these labels a screenreader user tabbing to a control will just be told a button is present, but not its purpose. For example,
    • “Next Faculty Member”, “Previous Faculty Member”;
    • the individual slide pickers may be generic, such as “Slide 2 of 5”, or a short descriptive text of the thumbnail.
    • In the slide selectors, the one currently selected is visually indicated, for instance by a solid dot, rather than a circle. This state must be made programmatically determinable using aria-selected=”true”.
    • The individual slide controls are presented visually as a list of items. This presentation must also be programmatically determinable. Mark-up the controls using a semantic <ul> list element.
    • Visually the next and previous buttons are grouped as a two item list. This presentation must also be programmatically determinable. Mark-up the two items using a semantic <ul> list element.

Automatically playing slideshow

As a general rule, carousels should not play automatically, as the movement may distract those with cognitive issues. If a carousel does play, then it should play for no more than 5 seconds, and have a "Pause" button.

Videos (1.2, 2.1.1, 3.2.1)

Video controls

Videos form a problematic area of accessibility, with many “off the shelf” solutions failing accessibility quite badly. The following will help ensure the player you use is as accessible as possible.

  • All video controls must be in the tab order, and be reachable and operable using only the keyboard.

  • All the controls must have visible focus indicators, so users can see which controls have received focus. See "Visible focus outlines" section.

  • All video controls must be marked up semantically with the correct roles: that is, buttons are marked with button roles (preferably <button>), sliders with slider roles, and so on.
  • All controls must have accessible text labels (these are usually text alternatives for symbols) that describe their purpose. Otherwise a screenreader user tabbing to a control will just be told a button is present, but not its purpose.
  • Where necessary the state of the control must be indicated – for instance, whether they are on or off, or what the current value of a slider is.

Automatically playing videos/Ambient videos

As a general rule, videos should not play automatically, as the movement may distract those with cognitive issues, and the sound will interfere with screenreader audio. If a video does play, then it should play for no more than 5 seconds if silent or 3 seconds if it includes audio.

Captions, audio descriptions and transcriptions

Some definitions:

  • Audio description: a sound track that describes any visual information and plays synched to speech and visuals.
  • Captions: subtitles appearing visually onscreen which reflects all that is said by actors, and describes important sounds and noises
  • Text transcription: full text alternative - transcription everything said, sound effects, visual information pertinent to understanding, and who has done the speaking. This text should either follow the video in the content; or a link to the transcription should be adjacent to the video.

To pass WCAG 2 level A:

  • Audio description OR text transcription is needed if there is visual information being conveyed in the video but not described in the audio.
  • Captions must be provided.
  • For silent video, there must be a short description of the video available, and if the video is informative, a text transcription provided.

To pass WCAG 2 level AA:

  • Must have audio description if visual information is not described in the standard audio track,
  • Does not need text description, but it is a nice to have.
  • Must have captions.

Loading or linking to a video player

Assuming an image such as the above acts as a control to load a video player, whether in place or in a modal, or is a link to a video page, then this control must be marked up with an accessible text label.

  • If it loads a video in place, or in a modal, the control must be a <button>.
  • If it links to another webpage with a video that starts automatically and has a player embedded there, the control must have a button role.
  • The play button is an information image. It must have an appropriate descriptive accessible text alternative, such as “Video:”, or “Load video”. See “Images" section.
  • Text positioned over background images must be presented accessibly with passable color contrast ratio.
  • If the image is decorative, it must be hidden from assistive technologies.

Choosing an accessible video player

Accessible video players are difficult to implement, or to find. Able player (https://ableplayer.github.io/ableplayer/) is one of the most accessible video players available. Alternately, the HTML5 YouTube video player is reasonably accessible, and has the advantage of being well known by users, meaning they are likely to know workarounds for any accessibility short falls.

Forms (3.3)

The following accessibility requirements are true for any form, and must be met here:

  • All individual form controls must have an accessible and programmatically associated label.
  • Label text must always be visible
  • Information must be provided to identify all required fields
  • Errors must be reported
  • Individual form controls with error messages must have those error messaged programmatically associated with them.
  • Color difference must not be the only visual clue to distinguish error messages from other content.
  • Captchas must be understandable and operable by keyboard only users, and by assistive technology users.
  • Labels and error messages must pass minimum color contrast requirements. This includes when visible labels are rendered as visible <label>s, or as placeholder text when the <label> is not visible.

Programmatically associated labels

All individual form fields must have a programmatically associated label. We highly recommend using the standard <label> element (even if they are not always visible).

<label for=”lastname>Last name”</label>
    <input id=”lastname” />

Label visibility

There must always be a visible label for each form control.

  • Label text is visible when the text box is empty (visually supplied by the placeholder text),
  • If the placeholder disappears when the textbox receives focus, then the <label> must be made visible
  • The <label> is visible when the text box contains text.

In short, a visual label must always be visible. This is because a screen magnification user might not see the label before they tab to the text box - if they have tabbed to a text box, and because the placeholder has disappeared there is no visible label, they will not know what input is expected of them.

Required fields

  • A description must be given on how required fields are indicated. It would be good if it were a little more descriptive than just “* required”.
  • For each required form control, the asterisk must be programmatically associated with the control – just make sure the asterisk is contained within the <label>, and not positioned outside it.
    <label for=”x”>Last name *</label>.
  • If possible, add a required and/or ARIA aria-required="required" attributes to the form control. This will ensure assistive technologies do inform users the field is required.

Reporting errors

When an user submits a form with errors, they must be informed that errors have occurred, and on which fields. Be as specific as possible – for instance, providing a list of all fields in error, where the fields are identified by naming their label values, and the error described. This can be best achieved by positioning a summary <ul> list of error messages at the beginning of the form with an appropriate heading, such as “Errors were found”. Positioning this after the form heading is a place that users can easily find.

<h2>Admissions form</h2>
    <h3>There were 4 errors found<h3>
    <p>Please correct the following errors</p>
    <ul>
        <li>First name missing</li>

If the page is not refreshed on submission (that is, page content is updated dynamically), move the focus to the error heading. Otherwise when a screenreader user submits the form with errors, it will appear as if nothing has happened.

Errors messages on individual fields

  • A field in error must have a programmatically associated error message, or error indicator.
  • A field in error must have a visible error indicator, that does not rely on color alone

Programmatically associated error message

Make sure that the error text is programmatically associated with the form control in error. Otherwise, when a visually impaired screenreader moves focus to this control, only the label “Phone number” may be read out, and they will not know there is an error; or they will have difficulties in finding fields in error when navigating through the form. The error message can be programmatically associated by adding the text inside the programmatically associated <label> element,

<label for=”phone”>
    <span class=”error”>Error: field missing</span>
    Phone number *
</label>

or by programmatically associating the text using the ARIA aria-describedby attribute. The screenreader will read out the label text, followed by the text referenced by the aria-describedby attribute,

<span id=”phone_error” class=”error”>
    Error: field missing
</span>
<label for=”phone”>
    Phone number *
</label>
<input id=”phone” aria-describedby=”phone_error” ... />

Does not rely on color alone

In the example, a message “Error: field missing” is given. This is good because the word “Error” gives a clear indication this is an error message without relying on the red color alone. Without this, the text would not necessarily be recognized as an error message text.

Note, an alternative to using plain text would be to use an error icon with an appropriate accessible text alternative.

Captchas

Captchas are very problematic in terms of accessibility, often relying on sight or sound. However, while the latest version Google's No CAPTCHA reCAPTCHA does present challenges for some disabled people though see below), with some reservations, our recommendation is to implement the latest version of Google No CAPTCHA reCAPTCHA.

Alternatives

There are a number of more accessible ways to protect against spam attacks, most of them discussed in the World Wide Web Consortium (W3C) Web Accessibility Initiative (WAI) Wiki page, "CAPTCHA Alternatives and thoughts” (https://www.w3.org/WAI/GL/wiki/Captcha_Alternatives_and_thoughts ).

We suggest that at least one of the alternatives could be used as the first step of a Multi-level approach. For instance:

  1. A "honey pot" trap is positioned between the first and second fieldset elements and given a label that would discourage screenreader users from selecting it; and
  2. If the checkbox is checked, a script function appends the reCAPTCHA to the end of the form.

This could potentially protect keyboard only users and screenreader users alike from encountering the CAPTCHA. However any multi level Turing test needs careful consideration and rigorous testing to ensure that the security of the form isn't compromised.

Infinite scrolling

Infinite scrolling (or loading) of a page is not addressed in any WCAG success criteria as yet. However, it should be noted that keyboard only users who are not using any assistive technologies might well have to tab through endless amounts of links before they can reach content beyond the listing, such as the page footer, and may well give up in frustration. We strongly recommend using pagination instead, with a reasonable amount of results per page.

The situation isn’t as problematic for screenreader users, if the list is marked up as an unordered list. They can jump to the end of the list element, and bypass the results if they wish. But we stress, this functionality is not available to disabled people who are not using an assistive technology.

This article recommends pagination, or a button that would load more results: https://simplyaccessible.com/article/infinite-scrolling/

Color contrast (1.4.1, 1.4.3)

The contrast between an outline color and the background color must meet the minimum 3:1 color contrast requirement.

If you are relying solely on changing the background color, of a button for instance, the color contrast between the focused and unfocussed background colors must be at least 3:1. The same applies if you are relying solely on a change of text color.

If you rely on the default browser focus styling, the background color on which it falls must also be the default background color of the browser (almost invariably white). Expecting a default focus indicator to be visible against a non-default background color will invariably make it difficult or impossible for sighted keyboard users to see which control has focus.
